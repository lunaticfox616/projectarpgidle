# Project ARPG Idle — AI Engineering Rulebook

이 문서는 이 저장소에서 작업하는 모든 AI Agent와 개발자가 따라야 하는 단일 룰북이다.
목표는 세션마다 반복 구현되는 코드, 숨은 결합, 거대 함수·파일, 의미 없는 테스트,
조용히 삼킨 오류를 방지하는 것이다.

`MUST`/`금지`는 예외 없는 요구사항이다. `SHOULD`/`권장`을 어겨야 한다면 변경 설명에
이유와 후속 정리 계획을 남긴다.

---

## 0. 작업 전 반드시 읽을 저장소 지도

현재 애플리케이션은 빌드 단계가 없는 브라우저 JavaScript 애플리케이션이다.
ES module이 아니라 `index.html`의 `<script>` 순서와 명시적으로 노출된 브라우저 전역을 사용한다.
이 구조를 모른 채 import/export 기반 프로젝트처럼 가정하지 않는다.

### 디렉터리 책임

| 경로 | 책임 | 의존 방향 |
| --- | --- | --- |
| `data/` | 게임 상수·정적 데이터·콘텐츠 정의 | 실행 로직에 의존하지 않는다 |
| `js/utils.js` | 도메인 비종속 공용 유틸리티와 안전한 전역 노출 도구 | `data/` 외의 기능 모듈에 의존하지 않는다 |
| `js/state.js` | 게임 상태의 기본 shape, 초기화, 상태 불변식 | 데이터와 유틸리티만 사용한다 |
| `js/save.js` | 저장/복원 경계와 직렬화 | 상태 shape와 유틸리티를 사용한다 |
| `js/items.js`, `js/skills.js`, `js/passives.js`, `js/core-cube.js` | 각 게임 도메인의 규칙 | 데이터·상태·유틸리티에 의존한다 |
| `js/combat-grid.js` | 8x8 그리드 전장 도메인(좌표·범위 패턴·이동·스폰 배치) | 데이터·상태에 의존하고 UI에 의존하지 않는다 |
| `js/combat.js` | 전투 및 진행 규칙 | 도메인 규칙에 의존하되 UI에 의존하지 않는다 |
| `js/canvas-*.js` | 캔버스 렌더링 어댑터 | 읽기 전용 도메인 결과를 받아 표현한다 |
| `js/ui.js`, `js/cosmos-atlas.js` | DOM/UI 조립과 사용자 이벤트 경계 | 도메인을 호출할 수 있지만 도메인이 이를 호출하게 만들지 않는다 |
| `js/main.js` | 최종 부트스트랩과 최상위 오케스트레이션 | 새 도메인 규칙을 구현하지 않는다 |
| `scripts/` | Node 기반 회귀·스모크 검사 | 프로덕션에서 불러오지 않는다 |

### 허용하는 기본 의존 방향

```text
data -> utils -> state/save -> domain(items/skills/passives/core-cube)
     -> combat -> rendering/UI -> main
```

- 화살표 오른쪽 모듈은 왼쪽 모듈을 사용할 수 있다.
- 반대 방향 의존, UI에서만 존재하는 함수에 대한 도메인 의존, 서로 호출하는 양방향 의존은 금지한다.
- 기존 전역 구조 때문에 의존성이 텍스트 import로 보이지 않을 수 있다. 새 의존을 추가할 때는
  `index.html`의 로드 순서와 생산자/소비자를 모두 확인한다.
- `index.html`의 스크립트 순서는 런타임 계약이다. 순서를 바꾸면 초기화, 저장 로드, 첫 렌더까지
  실제로 검증한다.

### 추가 사항
- 병렬화 가능한 도구 호출은 병렬 호출을 우선하라
- 파일을 병렬로 탐색하라. 순차 탐색이 필요하지 않은 이상 복수의 `rg` 또는 `Get-Content`는 한 번에 호출하라
- 수정 및 동작 검증은 모아서 한 번에 처리하는 것을 우선하라
- 모든 작은 함수마다 동작 테스트할 필요 없다. 기능 / 파일 단위로 한 번에 테스트하는 것을 우선 고려하라
- 매 코드 수정마다 `git diff`를 확인하는 것을 지양하라. 일반적으로 파일 전체 수정이 끝난 후 1번 확인하는 것으로 충분하다.
- 단, 작성한 코드가 동작을 보장하기 어려울 정도로 어렵고 복잡하거나, 다수의 함수 또는 파일에 걸쳐 작업해야 하는 경우에는 필요에 따라 해당 수정에 대해 즉시 검증을 진행할 수 있다.


## 1. 모든 작업의 필수 흐름

### 1.1 탐색 — 새 코드를 쓰기 전에

다음 순서로 기존 구현을 먼저 찾는다.

1. 대상 심볼과 유사한 도메인 용어를 `rg`로 검색한다.
2. `data/`, `js/utils.js`, `js/state.js`에서 기존 상수, helper, shape를 확인한다.
3. 함수 생산자뿐 아니라 모든 호출부와 `safeExposeGlobals`/`window` 노출부를 확인한다.
4. 관련 `scripts/smoke-*.js`를 읽어 이미 고정된 회귀 행동을 확인한다.
5. 기존 구현을 확장할 수 있으면 새 function/helper/shape를 만들지 않는다.

필수 탐색 예시:

```bash
rg -n "candidateName|domain term" data js scripts index.html
rg -n "safeExposeGlobals|window\\." js
```

**금지:** 검색 없이 편의상 `format*`, `normalize*`, `safe*`, `ensure*`, `get*` helper를 새로 만드는 것.
동일 개념은 이름이 달라도 중복이다.

### 1.2 설계 — 수정 전에

비자명한 변경은 코드를 쓰기 전에 아래를 짧게 정리한다.

- 변경할 행동과 변경하지 않을 행동
- 소유 모듈과 경계
- 입력, 출력, 상태 변경(side effect), 실패 방식
- 재사용할 기존 helper/type/shape
- 빈 입력, `null`/`undefined`, 경계값, 잘못된 저장 데이터, 실패 경로
- 새 의존의 방향과 스크립트 초기화 순서 영향
- 검증할 관찰 가능한 결과

두 개 이상의 도메인이 같은 shape를 소비한다면 shape의 소유자는 소비자가 아니라 가장 낮은 공통
계층이어야 한다. 정적 게임 정의는 `data/`, 런타임 상태 shape는 `js/state.js`, 도메인 비종속
변환은 `js/utils.js`를 단일 출처(SSOT)로 삼는다.

### 1.3 구현 — 가장 작은 응집된 변경

- 기존 코드를 대체해야 할 자리에는 기존 구현을 수정하거나 제거한다. 병렬 구현을 남기지 않는다.
- 함수는 한 가지 책임과 하나의 추상화 수준을 가진다.
- 상태 변경과 계산을 가능하면 분리한다. 계산 함수는 입력으로 결과를 결정하게 한다.
- 호출자가 알아야 하는 계약은 이름, JSDoc, 반환값 또는 명시적 오류로 드러낸다.
- 임시 호환 계층은 제거 조건과 범위를 주석으로 명시하지 않으면 만들지 않는다.
- 주석 처리 코드, 사용되지 않는 fallback, 죽은 분기는 커밋하지 않는다.

### 1.4 검증 — 구현과 함께

테스트는 구현 세부가 아니라 외부에서 관찰 가능한 행동을 검증한다.

- 정상 경로만 쓰지 말고 먼저 실패·경계 케이스를 고정한다.
- 반환값뿐 아니라 의도한 상태 변경과 의도하지 않은 상태 불변도 확인한다.
- 버그 수정은 수정 전 실패하고 수정 후 통과하는 회귀 검사를 추가한다.
- 관련 스모크 검사와 전체 스모크 검사를 실행한다.
- 브라우저 UI가 달라지면 실제 페이지에서 변경 흐름을 확인한다.

전체 스모크 검사:

```bash
for test_file in scripts/smoke-*.js; do node "$test_file" || exit 1; done
```

## 2. 결합도와 모듈 경계

### 2.1 숨은 결합 금지

다음은 모두 의존성이며 코드 리뷰에서 명시적으로 다룬다.

- `window` 또는 암묵적 전역에서 읽거나 쓰는 값
- 특정 `<script>`가 먼저 실행되어야만 존재하는 함수·상수
- import 없이 다른 파일의 함수 이름을 직접 호출하는 것
- 초기화 시점에 실행되는 상태 변경
- 공유 mutable object를 여러 모듈이 직접 수정하는 것
- DOM id/class와 도메인 규칙 사이의 암묵 계약
- 저장 데이터의 존재나 shape를 검증 없이 가정하는 것

새 전역은 기본적으로 금지한다. 레거시 인라인 HTML handler 또는 현재 런타임 조립상 전역 노출이
불가피하다면 다음을 모두 만족해야 한다.

1. 구현의 소유 모듈이 명확하다.
2. `safeExposeGlobals({ name })`를 사용한다. 임의의 `window.name = ...`를 추가하지 않는다.
3. 같은 이름의 기존 전역을 덮어쓰지 않는지 확인한다.
4. 소비자를 `rg`로 확인하고 테스트에서 공개 행동을 검증한다.
5. 상태 객체 자체보다 최소 기능의 함수를 노출한다.

### 2.2 순환 의존성과 fan-in/fan-out

- A가 B를 호출하고 B가 A를 호출하는 구조를 만들지 않는다.
- 순환이 필요해 보이면 공통 계산을 더 낮은 계층으로 이동하거나 상위 오케스트레이터가 양쪽을
  조정하게 한다.
- 한 함수가 여러 도메인의 상태를 직접 조립하면 해당 함수의 fan-out을 줄이고 오케스트레이션
  경계로 이동한다.
- 한 helper로 서로 무관한 호출이 계속 몰리면 이름만 공용인 god-helper인지 확인하고 도메인별로
  분리한다.
- 이벤트/콜백으로 순환을 숨기는 것도 순환 의존성으로 취급한다.

### 2.3 re-export와 공개 API

현재 저장소의 `safeExposeGlobals` 및 `window` 노출은 사실상 공개 API/re-export다.

- 사용되지 않는 전역 노출을 방치하지 않는다.
- 다른 파일이 다시 받아 같은 심볼을 재노출하지 않는다.
- 편의를 위한 배럴(barrel)식 일괄 노출을 만들지 않는다.
- 구현을 이동할 때 이전 노출과 새 노출을 동시에 영구 유지하지 않는다.
- 공개 심볼을 추가·삭제하면 모든 HTML handler와 JS 소비자를 함께 검사한다.

## 3. 함수·파일 복잡도 제한

### 3.1 새 코드의 상한

새 함수는 아래 한도를 지켜야 한다.

| 지표 | 상한 |
| --- | ---: |
| 함수 길이 | 60줄 |
| 중첩 깊이 | 3단계 |
| 순환 복잡도 | 10 |
| 함수 매개변수 | 5개 |
| 새 소스 파일 권장 크기 | 500줄 |

- 빈 줄과 단독 괄호를 빼서 숫자를 맞추는 식으로 우회하지 않는다.
- 긴 설정/정적 데이터는 함수 길이에서 예외일 수 있지만 `data/`가 소유해야 하는지 먼저 확인한다.
- 한도를 넘겨야 한다면 단순히 잘게 쪼개지 말고 책임 경계를 다시 설계한다.

### 3.2 기존 거대 파일에 대한 래칫 규칙

`js/ui.js`, `js/combat.js`, `js/passives.js` 등 기존 파일은 이미 위 한도를 넘는다. 이를 이유로
작업과 무관한 대규모 리팩터링을 섞지 않는다. 대신:

- 수정한 함수의 복잡도·길이·중첩을 이전보다 악화시키지 않는다.
- 새 독립 기능을 거대 파일 끝에 붙이지 않는다. 책임에 맞는 기존 파일 또는 새 응집 모듈을 쓴다.
- 긴 함수를 수정할 때 변경 범위 안에서 guard clause, 계산 추출, 중복 제거를 우선한다.
- 파일 분리는 로드 순서, 전역 공개 API, 상태 초기화 계약을 테스트로 고정한 뒤 별도 변경으로 한다.

### 3.3 중첩 평탄화

3단계 이상의 중첩이 생기면 다음 순서로 줄인다.

1. invalid/empty/failure 조건을 guard clause로 반환한다.
2. 루프 내부 조건은 `continue`로 분리한다.
3. 서로 다른 책임을 이름 있는 함수로 추출한다.
4. 복잡한 boolean은 의미 있는 predicate로 이름 붙인다.
5. 중첩 callback 대신 명시적인 단계 함수를 사용한다.

삼항 연산자를 중첩해 줄 수만 줄이는 것은 평탄화가 아니다.

## 4. 오류 처리와 방어 코드

### 4.1 오류를 조용히 삼키지 않는다

금지 예시:

```js
try {
    riskyOperation();
} catch (error) {
    // ignore
}

const value = maybeBroken() || defaultValue;
```

오류는 다음 경계 중 한 곳에서만 처리한다.

- 사용자 입력/DOM 이벤트 경계
- 저장 데이터 역직렬화·마이그레이션 경계
- 네트워크/클라우드 저장 경계
- 애플리케이션 최상위 부트스트랩 경계

낮은 계층은 복구할 수 없는 오류를 숨기지 않고 호출자에게 전달한다. 경계에서 오류를 처리할 때는:

1. 어떤 작업이 실패했는지 문맥을 보존한다.
2. 사용자 또는 로그에 관찰 가능한 신호를 남긴다.
3. 상태를 부분 갱신한 채 성공처럼 진행하지 않는다.
4. fallback이 제품 요구사항이라면 언제 적용되는지 테스트한다.

### 4.2 중복 방어 금지

- 같은 `null`/shape 검사를 호출부마다 반복하지 않는다.
- 외부/저장 입력은 진입 경계에서 한 번 정규화하고 내부에는 정규화된 계약을 전달한다.
- 내부 불변식 위반까지 기본값으로 덮지 않는다. 빠르게 실패시켜 원인을 드러낸다.
- `ensure*`, `safe*`, `normalize*` 함수는 명확한 경계 소유자가 있을 때만 만든다.
- 테스트를 통과시키기 위해 프로덕션 코드에 무의미한 `if (!x) return`을 추가하지 않는다.

### 4.3 `try/catch` 기준

`catch`는 아래 중 하나를 수행해야 한다.

- 의미 있는 복구
- 더 구체적인 문맥을 포함한 오류로 재전파
- 명시적인 사용자 오류 상태로 변환
- 로깅 후 안전한 트랜잭션/상태 롤백

빈 `catch`, 로그도 없이 기본값 반환, 모든 예외를 동일 처리하는 광범위한 `catch`는 금지한다.

## 5. 타입과 데이터 shape

이 저장소는 JavaScript이므로 TypeScript의 `strict`를 직접 적용할 수 없다. 대신 변경하는 공개 경계와
복잡한 shape에는 JSDoc을 사용해 암묵 계약을 드러낸다.

```js
/**
 * @param {Readonly<PlayerStats>} stats
 * @param {number} elapsedMs
 * @returns {CombatTickResult}
 */
function calculateCombatTick(stats, elapsedMs) {
    // ...
}
```

- `@typedef`는 해당 shape의 소유 모듈 한 곳에만 둔다.
- `Object`, `Function`, `*`, `any`처럼 계약을 숨기는 타입은 금지한다.
- TypeScript를 도입할 경우 `strict: true`를 유지하고 `as any`, `as unknown as T`, 광범위한
  non-null assertion으로 오류를 우회하지 않는다.
- 선택 필드, nullable 필드, 단위(ms/초, 비율/퍼센트), 식별자 종류를 명시한다.
- 저장 shape 변경은 기본값, 이전 버전 마이그레이션, 손상 데이터 실패 경로를 함께 검증한다.
- 같은 개념에 서로 다른 필드 이름이나 단위를 만들지 않는다.

## 6. 테스트 룰

### 6.1 행동을 검증한다

좋은 테스트는 입력/행동/관찰 가능한 결과를 검증한다.

- 반환값
- 게임 상태의 의도한 변화
- 저장 결과
- DOM에 표시되는 사용자 결과
- 오류 또는 실패 상태
- 호출하지 않아야 할 외부 side effect

다음만 확인하는 테스트는 금지한다.

- 소스 문자열에 함수 이름이 존재하는지
- 특정 내부 변수명이나 구현 문장이 있는지
- 정규식으로 코드 모양만 맞는지
- 함수 본문을 실행하지 않고 export/global 이름만 있는지
- 커버리지 수치만 올리지만 실패 가능 행동을 검증하지 않는지

정적 문자열 검사는 자산 경로, HTML 연결, 금지 패턴처럼 정적 구조 자체가 요구사항일 때만 허용하며,
그 검사를 행동 테스트의 대체물로 사용하지 않는다.

### 6.2 엣지 케이스를 먼저 고정한다

관련되는 항목을 구현 전에 테스트 목록으로 만든다.

- 빈 배열, 빈 문자열, 빈 객체
- `null`, `undefined`, 누락 필드
- 0, 음수, 최솟값/최댓값, 한계 바로 전·후
- 중복 id와 존재하지 않는 id
- 손상되거나 이전 버전인 저장 데이터
- 작업 중 실패하여 부분 상태만 남을 가능성
- 타이머/이벤트가 연속 또는 중복 실행되는 상황
- 동시에 발생하는 저장·UI 갱신 등 경쟁 상태
- 외부 API/스토리지 실패와 지연

모든 함수에 기계적으로 모든 케이스를 추가하라는 뜻은 아니다. 계약상 가능한 실패를 선택하고 왜
관련 없는지 설명할 수 있어야 한다.

### 6.3 mock은 외부 경계에서만

mock/stub/fake를 허용하는 경계:

- 시간과 난수
- 네트워크와 Supabase
- 브라우저 스토리지
- DOM/canvas처럼 Node 검사에서 제공되지 않는 플랫폼 API

금지:

- 같은 도메인의 내부 helper를 mock하여 구현 순서를 고정하는 것
- private 함수 호출 횟수를 테스트하는 것
- 실제 상태 전이를 우회하고 성공값만 반환하는 mock
- 많은 mock 설정이 있어야만 통과하는 단위 테스트

내부 모듈은 실제 구현을 함께 실행하는 작은 통합 테스트를 우선한다.

### 6.4 side effect 검사

상태 변경 함수의 테스트는 최소한 다음을 확인한다.

1. 변경되어야 하는 필드가 정확히 변경되었다.
2. 관련 없는 필드는 유지되었다.
3. 실패 시 부분 변경이 남지 않았다.
4. 저장, 로그, DOM, 이벤트 같은 외부 효과가 필요한 횟수만 발생했다.
5. 같은 이벤트가 재실행될 수 있다면 중복 적용 여부를 확인했다.

## 7. 정적 분석과 CI 강제 정책

도구를 추가하거나 갱신할 때 다음 규칙을 기계적으로 강제한다. 설정을 완화해 테스트를 통과시키지
않고 위반 코드를 고친다.

### JavaScript lint 최소 정책

- 복잡도: `complexity: 10`
- 중첩: `max-depth: 3`
- 함수 길이: `max-lines-per-function: 60`
- 매개변수: `max-params: 5`
- `no-unused-vars`, `no-undef`, `no-unreachable`
- 빈 블록 및 빈 `catch` 금지
- 암묵적 전역 금지

기존 거대 함수 때문에 저장소 전체 적용이 당장 불가능하면 변경 파일/신규 코드부터 적용하고
baseline을 기록해 위반 수가 증가하지 않는 래칫으로 운용한다. 새로운 disable 주석은 규칙명,
구체적 이유, 제거 조건 없이는 금지한다.

### 의존성 검사 최소 정책

CI는 최종적으로 아래 위반 시 실패해야 한다.

- 순환 의존성
- `data/` 또는 저수준 도메인에서 UI 계층으로 향하는 의존
- 허용 목록에 없는 새 `window` 전역
- 중복 공개 심볼
- 로드 순서상 생산자보다 먼저 실행되는 소비자

현재의 classic script 구조에서는 import graph 도구만으로 충분하지 않다. `index.html`의 script 순서와
`safeExposeGlobals` 생산자/소비자도 검사 대상에 포함한다.

## 8. 변경 리뷰 체크리스트

### 8.1 설계 리뷰

- [ ] 모듈/레이어 경계와 의존 방향을 먼저 정했는가?
- [ ] 공용 utility/type/shape의 SSOT를 확인했는가?
- [ ] 새 파일의 위치가 도메인과 위계를 드러내는가?
- [ ] 새 전역 또는 초기화 순서 의존을 피했는가?
- [ ] 입력, 출력, side effect, 실패 방식이 명시적인가?

### 8.2 구현 리뷰

- [ ] 새 helper/function/shape를 만들기 전에 기존 구현을 검색했는가?
- [ ] 기존 코드를 대체해야 할 자리에 중복 구현이 생기지 않았는가?
- [ ] 죽은 코드와 주석 처리 코드가 남지 않았는가?
- [ ] 함수가 60줄, 깊이 3, 복잡도 10 한도 안에 있는가?
- [ ] guard clause로 불필요한 중첩을 줄였는가?
- [ ] 도메인 계산과 UI/저장 side effect가 분리되어 있는가?
- [ ] 오류를 삼키거나 불명확한 fallback으로 덮지 않았는가?
- [ ] 호출부마다 같은 방어 로직을 복제하지 않았는가?

### 8.3 결합도 리뷰

- [ ] 암묵 전역, 숨은 import, 초기화 순서 의존이 추가되지 않았는가?
- [ ] 전역 상태를 여러 모듈이 무제한으로 수정하지 않는가?
- [ ] 순환 또는 양방향 의존이 없는가?
- [ ] fan-in/fan-out이 한 함수나 파일에 과도하게 몰리지 않았는가?
- [ ] 사용되지 않는 전역 공개 API/re-export를 정리했는가?
- [ ] `index.html` script 순서 변경의 런타임 영향을 확인했는가?

### 8.4 테스트 리뷰

- [ ] 소스 문자열이 아니라 출력 행동을 검증하는가?
- [ ] 정상 경로보다 실패·경계 경로를 먼저 고려했는가?
- [ ] 빈 입력, nullish 값, 경계값, 실패 경로 중 관련 항목이 고정되었는가?
- [ ] 의도한 side effect와 의도하지 않은 side effect를 모두 확인하는가?
- [ ] mock이 외부 경계에만 있는가?
- [ ] 테스트를 위해 프로덕션 방어 코드를 추가하지 않았는가?
- [ ] 관련 검사와 전체 스모크 검사가 통과하는가?

## 9. 완료 정의(Definition of Done)

다음을 모두 만족해야 작업이 완료된다.

1. 요구 행동이 구현되었고 범위 밖 행동이 바뀌지 않았다.
2. 중복 helper/type/shape/global을 만들지 않았다.
3. 새 의존과 side effect가 올바른 계층에 있고 숨겨져 있지 않다.
4. 오류 처리 책임이 경계 한 곳에 있다.
5. 함수·파일 복잡도가 한도를 지키거나 기존 위반을 악화시키지 않았다.
6. 행동 기반 회귀 테스트가 정상·경계·실패 경로를 적절히 고정한다.
7. 관련 스모크 및 전체 스모크 검사를 실행했다.
8. 변경 설명에 위험, 의존성/초기화 영향, 검증 명령을 기록했다.

속도 때문에 이 룰북을 우회하지 않는다. AI Agent의 세션 기억은 연속적이지 않으므로, 다음 세션이
추론해야만 알 수 있는 계약은 존재하지 않는 계약으로 간주한다. 계약은 코드 구조, JSDoc, 테스트,
이 룰북 중 적절한 위치에 남긴다.
